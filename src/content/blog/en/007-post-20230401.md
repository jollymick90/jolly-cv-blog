# Keycloak Customize Theme: Frontend Challenge

You've just installed and configured Keycloak in your infrastructure to manage authentication and authorization. Everything works. Users can log in, tokens are issued, roles are assigned correctly.

But then the product team opens the login page and asks: *"Why does it look like it's from 2005?"*

Fair point. Let's fix it.

---

## What is a Keycloak Theme

Keycloak controls multiple UI surfaces: the login page, the account management console, the admin UI, registration and password-reset forms, and transactional emails. Each of these is a **theme**.

There are five theme types:

| Type | What it controls |
|------|-----------------|
| `login` | Login, registration, password reset, OTP pages |
| `account` | User account self-service portal |
| `admin` | The Keycloak admin console |
| `email` | Transactional emails (verification, reset link) |
| `welcome` | The Keycloak welcome page |

For most applications, you only need to customize `login` — it's the surface your end users actually see.

---

## How Themes Work

Keycloak themes are built with **FreeMarker** (`.ftl` files), CSS, and static assets. They live inside the Keycloak server at:

```
$KEYCLOAK_HOME/themes/
    ├── base/        ← the base theme with all templates
    ├── keycloak/    ← default styling (extends base)
    └── your-theme/  ← your custom theme
```

The resolution is layered: Keycloak first looks in your theme, then falls back to the parent theme, then to `base`. You only need to override what you actually want to change.

## Creating Your Custom Theme

### Step 1 — Folder structure

Create the folder for your theme:

```
themes/
└── my-app/
    └── login/
        ├── theme.properties
        ├── resources/
        │   ├── css/
        │   │   └── styles.css
        │   └── img/
        │       └── logo.png
        └── templates/          ← optional, only if overriding FTL
            └── login.ftl
```

### Step 2 — theme.properties

This is the entry point. Declare what you extend and which resources to load:

```properties
parent=keycloak
import=common/keycloak

styles=css/styles.css
```

Setting `parent=keycloak` means you inherit all default templates and only override what you explicitly include.

### Step 3 — Override CSS

The quickest path to a custom look. Keycloak's default stylesheet exposes CSS variables you can override:

```css
/* themes/my-app/login/resources/css/styles.css */

:root {
  --pf-global--FontFamily--sans-serif: 'Inter', sans-serif;
  --pf-global--Color--100: #1a1a2e;
  --pf-global--primary-color--100: #0046c9;
  --pf-global--primary-color--200: #003bb0;
}

.login-pf body {
  background: #f0f4ff;
}

.card-pf {
  border-radius: 0;
  box-shadow: 0 2px 12px rgba(0, 70, 201, 0.08);
}

#kc-header-wrapper {
  background-color: #0046c9;
  padding: 1.5rem;
}
```

### Step 4 — Override a template

For deeper changes (restructuring the layout, adding a custom logo block, changing form fields order), you copy the FreeMarker template from the `base` theme and modify it.

Get the original from:
```
$KEYCLOAK_HOME/themes/base/login/login.ftl
```

Then copy it to your theme and edit:

```freemarker
<#-- themes/my-app/login/templates/login.ftl -->
<#import "template.ftl" as layout>

<@layout.registrationLayout displayMessage=!messagesPerField.existsError('username','password') displayInfo=realm.password && realm.registrationAllowed && !registrationDisabled??; section>
    <#if section = "header">
        <img src="${url.resourcesPath}/img/logo.png" alt="${realm.displayName}" class="logo" />
    <#elseif section = "form">
        <form id="kc-form-login" action="${url.loginAction}" method="post">
            <div class="${properties.kcFormGroupClass!}">
                <label for="username">${msg("usernameOrEmail")}</label>
                <input type="text" id="username" name="username" value="${(login.username!'')}" autofocus />
            </div>
            <div class="${properties.kcFormGroupClass!}">
                <label for="password">${msg("password")}</label>
                <input type="password" id="password" name="password" />
            </div>
            <input type="submit" value="${msg('doLogIn')}" />
        </form>
    </#if>
</@layout.registrationLayout>
```

## Activating the Theme

### Via Admin Console

1. Open **Keycloak Admin Console**
2. Select your realm
3. Go to **Realm Settings → Themes**
4. Set **Login Theme** to `my-app`
5. Save

### Via Docker / Environment Variable

If you're running Keycloak in Docker, mount your theme folder and set the theme at startup:

```yaml
services:
  keycloak:
    image: quay.io/keycloak/keycloak:23.0
    volumes:
      - ./themes/my-app:/opt/keycloak/themes/my-app
    environment:
      KC_SPI_THEME_DEFAULT: my-app
```

Or configure it per-realm via the Admin REST API if you need it in a CI/CD pipeline.

## Useful Tips

**Disable theme caching in dev mode.** By default Keycloak caches themes. During development, pass:
```
--spi-theme-static-max-age=-1
--spi-theme-cache-themes=false
--spi-theme-cache-templates=false
```

**Use the message catalog.** Keycloak has i18n built in. Override messages in:
```
themes/my-app/login/messages/messages_en.properties
```

```properties
loginTitle=Welcome back
usernameOrEmail=Email address
doLogIn=Sign in
```

**Keep templates thin.** Don't fight FreeMarker — override only what is necessary. The less you copy from `base`, the easier your upgrades will be when Keycloak releases a new version.

**Test all flows.** A login page is not just the happy path. Test: login error, wrong password, forgot password, registration, OTP, account verified email. Each has its own template.

---

## Conclusion

Customizing a Keycloak theme has a learning curve — mostly around FreeMarker syntax and understanding which template controls which screen. But the layer-based inheritance model is solid: once you get the folder structure right, you can go from the default Keycloak look to a fully branded experience with only CSS in most cases.

The key discipline is **minimalism**: override only what you must, inherit the rest. Your future self will thank you at upgrade time.

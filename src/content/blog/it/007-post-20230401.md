# Keycloak Customize Theme: Frontend Challenge

Hai installato e configurato Keycloak nella tua infrastruttura per gestire autenticazione e autorizzazione. Tutto funziona. Gli utenti accedono, i token vengono rilasciati, i ruoli sono assegnati correttamente.

Poi il team di prodotto apre la pagina di login e chiede: *"Perché sembra uscita dal 2005?"*

Giusto. Risolviamo.

---

## Cosa sono i Keycloak Theme

Keycloak gestisce diverse interfacce utente: la pagina di login, la console di gestione account, l'UI di amministrazione, i form di registrazione e reset password, e le email transazionali. Ognuna di queste è un **theme**.

Esistono cinque tipi di tema:

| Tipo | Cosa controlla |
|------|----------------|
| `login` | Login, registrazione, reset password, OTP |
| `account` | Portale self-service dell'utente |
| `admin` | La console di amministrazione Keycloak |
| `email` | Email transazionali (verifica, link di reset) |
| `welcome` | La pagina di benvenuto di Keycloak |

Per la maggior parte delle applicazioni, è sufficiente personalizzare il `login` — è l'unica superficie che i tuoi utenti finali vedono davvero.

---

## Come Funzionano i Temi

I temi Keycloak sono costruiti con **FreeMarker** (file `.ftl`), CSS e asset statici. Risiedono all'interno del server Keycloak in:

```
$KEYCLOAK_HOME/themes/
    ├── base/        ← il tema base con tutti i template
    ├── keycloak/    ← stile di default (estende base)
    └── il-tuo-tema/ ← il tuo tema custom
```

La risoluzione è a livelli: Keycloak cerca prima nel tuo tema, poi nel tema padre, poi in `base`. Devi sovrascrivere solo ciò che vuoi cambiare.

## Creare il Tuo Tema Custom

### Step 1 — Struttura delle cartelle

Crea la cartella per il tuo tema:

```
themes/
└── my-app/
    └── login/
        ├── theme.properties
        ├── resources/
        │   ├── css/
        │   │   └── styles.css
        │   └── img/
        │       └── logo.png
        └── templates/          ← opzionale, solo se sovrascrivi FTL
            └── login.ftl
```

### Step 2 — theme.properties

È il punto di ingresso. Dichiari da cosa estendi e quali risorse caricare:

```properties
parent=keycloak
import=common/keycloak

styles=css/styles.css
```

Impostando `parent=keycloak` erediti tutti i template di default e sovrascrivi solo ciò che includi esplicitamente.

### Step 3 — Sovrascrivere il CSS

Il percorso più rapido per un look personalizzato. Il foglio di stile di default di Keycloak espone variabili CSS che puoi sovrascrivere:

```css
/* themes/my-app/login/resources/css/styles.css */

:root {
  --pf-global--FontFamily--sans-serif: 'Inter', sans-serif;
  --pf-global--Color--100: #1a1a2e;
  --pf-global--primary-color--100: #0046c9;
  --pf-global--primary-color--200: #003bb0;
}

.login-pf body {
  background: #f0f4ff;
}

.card-pf {
  border-radius: 0;
  box-shadow: 0 2px 12px rgba(0, 70, 201, 0.08);
}

#kc-header-wrapper {
  background-color: #0046c9;
  padding: 1.5rem;
}
```

### Step 4 — Sovrascrivere un template

Per modifiche più profonde (ristrutturare il layout, aggiungere un blocco logo custom, cambiare l'ordine dei campi del form), copi il template FreeMarker dal tema `base` e lo modifichi.

Prendilo da:
```
$KEYCLOAK_HOME/themes/base/login/login.ftl
```

Poi copialo nel tuo tema e modifica:

```freemarker
<#-- themes/my-app/login/templates/login.ftl -->
<#import "template.ftl" as layout>

<@layout.registrationLayout displayMessage=!messagesPerField.existsError('username','password') displayInfo=realm.password && realm.registrationAllowed && !registrationDisabled??; section>
    <#if section = "header">
        <img src="${url.resourcesPath}/img/logo.png" alt="${realm.displayName}" class="logo" />
    <#elseif section = "form">
        <form id="kc-form-login" action="${url.loginAction}" method="post">
            <div class="${properties.kcFormGroupClass!}">
                <label for="username">${msg("usernameOrEmail")}</label>
                <input type="text" id="username" name="username" value="${(login.username!'')}" autofocus />
            </div>
            <div class="${properties.kcFormGroupClass!}">
                <label for="password">${msg("password")}</label>
                <input type="password" id="password" name="password" />
            </div>
            <input type="submit" value="${msg('doLogIn')}" />
        </form>
    </#if>
</@layout.registrationLayout>
```

## Attivare il Tema

### Dalla Admin Console

1. Apri la **Keycloak Admin Console**
2. Seleziona il tuo realm
3. Vai su **Realm Settings → Themes**
4. Imposta **Login Theme** su `my-app`
5. Salva

### Via Docker / Variabile d'ambiente

Se stai eseguendo Keycloak in Docker, monta la cartella del tema e imposta il tema all'avvio:

```yaml
services:
  keycloak:
    image: quay.io/keycloak/keycloak:23.0
    volumes:
      - ./themes/my-app:/opt/keycloak/themes/my-app
    environment:
      KC_SPI_THEME_DEFAULT: my-app
```

Oppure configuralo per realm tramite la Admin REST API per pipeline CI/CD.

## Consigli Pratici

**Disabilita la cache dei temi in sviluppo.** Di default Keycloak fa cache dei temi. Durante lo sviluppo, passa queste opzioni:
```
--spi-theme-static-max-age=-1
--spi-theme-cache-themes=false
--spi-theme-cache-templates=false
```

**Usa il catalogo dei messaggi.** Keycloak ha l'i18n integrato. Sovrascrivi i messaggi in:
```
themes/my-app/login/messages/messages_it.properties
```

```properties
loginTitle=Bentornato
usernameOrEmail=Indirizzo email
doLogIn=Accedi
```

**Mantieni i template snelli.** Non combattere con FreeMarker — sovrascrivi solo il necessario. Meno copi da `base`, più facili saranno gli aggiornamenti quando Keycloak rilascerà una nuova versione.

**Testa tutti i flussi.** Una pagina di login non è solo l'happy path. Testa: errore di login, password errata, password dimenticata, registrazione, OTP, email di verifica account. Ogni flusso ha il suo template.

---

## Conclusione

Personalizzare un tema Keycloak ha una curva di apprendimento — principalmente sulla sintassi FreeMarker e sul capire quale template controlla quale schermata. Ma il modello di ereditarietà a livelli è solido: una volta che hai la struttura delle cartelle giusta, nella maggior parte dei casi puoi passare dal look predefinito a un'esperienza completamente brandizzata con solo CSS.

La disciplina chiave è il **minimalismo**: sovrascrivi solo ciò che è strettamente necessario, eredita il resto. Il tuo futuro self ti ringrazierà al momento degli upgrade.
